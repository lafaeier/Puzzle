# 项目开发章程与大纲

## 一、项目总览

| 维度 | 内容 |
|---|---|
| 游戏类型 | 第一人称解谜类AVG |
| 引擎 | Unreal Engine 5.7 |
| 模块名 | Puzzle（单模块） |
| 目标平台 | Windows |
| 核心玩法 | WASD移动 + 鼠标视角 + F键交互，收集线索/物品，限时逃离关卡 |
| 世界观 | 主角进入梦境，通过解谜找回遗忘的美好与执念，不同选择导向不同结局 |
| 美术风格 | 偏写实室内场景 |

### 关卡规划

| 关卡 | 场景 | 简述 |
|---|---|---|
| Level_Bedroom | 卧室 | 起始关卡，找到客厅钥匙，开门进入客厅 |
| Level_LivingRoom | 客厅 | 目标关卡，可探索支线道具，可返程回卧室 |
| （可选）第三关 | 待定 | 如有时间实现 |

### 操作映射

| 按键 | 功能 |
|---|---|
| W/A/S/D | 移动 |
| 鼠标移动 | 旋转视角 |
| F | 与物体交互（射线检测） |
| Tab | 打开背包界面 |
| Esc | 暂停菜单（保存/退出） |

## 二、技术架构

### 2.1 分层架构

```
┌─────────────────────────────────┐
│     关卡 (.umap)                │  ← 场景层：摆放Actor实例，配置Transform
├─────────────────────────────────┤
│     蓝图子类 (.uasset)           │  ← 桥接层：套模型/材质，配置UPROPERTY参数
├─────────────────────────────────┤
│     C++ 基类 (.h/.cpp)          │  ← 逻辑层：定义接口、状态管理、核心逻辑
├─────────────────────────────────┤
│     UE5 引擎层                   │  ← 不可变：渲染、物理、EnhancedInput等
└─────────────────────────────────┘
```

### 2.4 关键数据流

```
玩家按F → LineTraceByChannel
              ↓
      命中Actor是否实现了 IInteractable？
         ↓ 是                    ↓ 否
  Interact_Implementation()    静默忽略
         ↓
  ┌──────┼──────┬──────────────┐
  ↓      ↓      ↓              ↓
储物柜  钥匙   门触发器       其他交互物
  ↓      ↓      ↓
  ├→ 检查全局状态是否已开      校验GameInstance中是否有钥匙
  ├→ 未开→生成钥匙道具        有→OpenLevel切换关卡
  ├→ 更新RoomGameInstance      无→显示文字提示"需要钥匙"
  └→ 播放柜门动画
                  ↓
              钥匙道具
              ├→ 调用InventoryComponent::AddItem()
              ├→ 更新RoomGameInstance::bHasKeyToLivingRoom = true
              └→ Destroy() 销毁自身
```

**核心规则**：所有解谜进度、道具持有状态仅存储在 `URoomGameInstance` 中，关卡切换不重置。

## 三、开发章程

### 3.1 代码规范

**命名约定：**
- C++类：遵循UE前缀规则 — `A`开头= Actor, `U`开头= UObject/Component, `I`开头= Interface
- 成员变量：小驼峰，bool以`b`开头（如 `bHasKey`, `bOpened`）
- 函数：大驼峰，暴露给蓝图的用动词开头（`Interact`, `AddItem`, `OpenDoor`）
- 文件名 = 类名（不含前缀）：`StorageCabinet.h` 包含 `AStorageCabinet` 类

**UPROPERTY规范：**
- 需要在编辑器/蓝图调整的参数 → `EditAnywhere, BlueprintReadWrite`
- 仅蓝图读取 → `BlueprintReadOnly`
- 需要保存到存档 → 加 `SaveGame` 标记
- 始终加 `Category = "xxx"` 组织属性面板

**逻辑归属原则：**
- 复杂逻辑 → C++ 实现
- 蓝图 → 只做：模型/材质赋值、UPROPERTY参数配置、简单事件连线、动画/特效触发
- 不把核心判断逻辑写在蓝图里（无法diff，难以排查）

### 3.2 编译与构建

```bash
# 编辑器目标（开发阶段使用）
Engine\Build\BatchFiles\Build.bat PuzzleEditor Win64 Development -Project="E:\subject\UE\Puzzle\Puzzle.uproject"

# 游戏目标（打包使用）
Engine\Build\BatchFiles\Build.bat Puzzle Win64 Development -Project="E:\subject\UE\Puzzle\Puzzle.uproject"
```

- 每次修改 `.h` 文件后必须重新编译
- 新增 `.h/.cpp` 文件后，确认 `Puzzle.Build.cs` 中包含所需模块依赖
- 编译失败 → 不要盲目重试，先读错误信息
- `.generated.h` 文件由 UHT 自动生成，不手动修改

### 3.3 开发顺序（关键）

**严格按此顺序开发，后续依赖前置：**

```
阶段1：数据层 ✅ 已完成 (2026-06-14)
  → RoomGameInstance（全局状态容器）✅
  → RoomSaveGame（存档结构体）✅
  → RoomGameModeBase（基础GameMode）✅

阶段2：角色层 ← 当前阶段
  → IInteractable（交互接口）
  → RoomPlayerCharacter（角色+F键射线交互）
  → RoomPlayerController（输入配置）
  → InventoryComponent（背包组件）

阶段3：交互物层
  → InteractableActor（可交互物基类）
  → PickupItem（可拾取道具基类）
  → StorageCabinet（储物柜）

阶段4：关卡切换
  → DoorTrigger（门触发器）

阶段5：场景搭建
  → Blueprint子类创建
  → 关卡布局（摆放Actor）
  → 测试与调试

阶段6：支线（如有时间）
  → PasswordDrawer、LightSocket、UI组件
```

### 3.4 场景与代码的交接规则

- C++ 类必须先于 Blueprint 创建：先写 `.h/.cpp` → 编译通过 → 再创建 Blueprint 子类
- 接口一旦定版，修改需通知所有依赖方
- 蓝图不实现核心逻辑，只做视觉表现和参数配置
- 每次 C++ 新增 `UPROPERTY(EditAnywhere)` 后，蓝图中重新检查该属性的默认值

## 四、场景搭建与代码对接点

这是整个项目开发中最关键的环节。以下列出每个系统在代码层和场景层的具体对接方式。

### 4.1 玩家角色（对接点 #1）

```
代码层                         场景层
────────────────────────────────────────────────
ARoomPlayerCharacter            BP_FirstPersonPlayer (蓝图子类)
├─ UPROPERTY InteractRange     蓝图设为 300cm（交互距离）
├─ UPROPERTY InteractAction    绑定 IA_Interact（F键）
├─ UPROPERTY InventoryComp     自动创建背包组件
├─ USkeletalMesh (手臂)        蓝图挂载第一人称手臂模型
└─ UCameraComponent            蓝图调整相机高度到人眼位置(170cm)
                                拖入关卡 Level_Bedroom 的 PlayerStart 处
```

### 4.2 交互接口（对接点 #2）

```
代码层                         场景层
────────────────────────────────────────────────
IInteractable (C++接口)        
├─ 纯虚函数 Interact_Implementation()
├─ 纯虚函数 GetInteractPrompt() → 返回FString提示文字
│                                    
实现此接口的类，在场景中的表现：
  - 玩家靠近时，十字准星变化或出现文字提示
  - 准星对准可交互物时，显示 GetInteractPrompt() 返回的文本
  - 按F键 → 射线命中 → 通过接口指针调用 Interact_Implementation()
  - 未命中任何可交互物 → 静默忽略
```

### 4.3 储物柜（对接点 #3）★主线核心

```
C++ 定义                        蓝图配置                      关卡实例
─────────────────────────────────────────────────────────────────────

├─ bOpened = false              模型: CabinetMesh.fbx        位置: 卧室墙角(300, 500, 0)
├─ SpawnItemClass               SpawnItemClass → BP_Item_Key 旋转: 面向玩家出生点
│  (TSubclassOf<APickupItem>)                                碰撞: 正面放置BoxCollision
├─ SpawnTransform               动画: 柜门open骨骼动画
│  (FTransform)                 特效: 灰尘粒子（可选）
├─ Interact_Implementation()    音效: 柜门吱呀声
│  ├─ 检查 bOpened                                            
│  ├─ 生成 SpawnItemClass
│  ├─ bOpened = true
│  └─ 写 GameInstance
└─ GetInteractPrompt()
   ├─ 未开: "按F打开柜子"
   └─ 已开: "柜子已打开"
```


### 5.2 操作规范

| 规则 | 原因 |
|---|---|
| `.uasset` 修改前确认无他人锁定 | 二进制文件无法合并 |
| C++ 头文件修改后，编译通过再推送 | 编译失败的代码阻塞所有人 |
| 新C++类涉及新模块依赖时，同步更新 `Puzzle.Build.cs` | 缺少依赖编译不过 |
| 每完成一个小功能点就commit，不攒大提交 | 二进制冲突的代价随时间指数增长 |
| commit message 写清"改了什么资源、为什么改" | 二进制无法diff，message是唯一信息源 |
| 复杂逻辑写C++，蓝图仅做配置和视觉 | C++可diff/merge，蓝图不能 |

### 5.3 关卡文件的协作方式

每个 `.umap` 关卡文件在任一时刻只由一个人编辑。多人同时编辑同一关卡几乎必然产生不可合并的冲突。应对策略：

- **主关卡专人负责**：Level_Bedroom 由一人主搭，Level_LivingRoom 由另一人主搭
- **通过蓝图间接贡献**：第三人做的交互物蓝图（如 BP_Prop_NewItem）完成后，由关卡负责人拖入关卡，而不是第三人直接改关卡
- **关卡完成后锁定**：关卡定型后，后续调整通过蓝图UPROPERTY参数完成，不再频繁修改.umap
